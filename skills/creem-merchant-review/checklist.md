# Creem 过审自查表（Chrome 扩展 + 官网 + 订阅 形态）

> 整理依据：Creem 官方 Account Reviews 文档及 Merchant Terms（截至 2026-06）。具体以提交时 Creem 后台与官方文档为准。
> 核心心法：Creem 是 MoR，替商户对终端用户兜底退款/拒付，审核的是"商户与产品可不可信、出事能不能找到你、用户能不能自助解决"。

判定图例：✅ 已就绪 / ⚠️ 有但不到位 / ❌ 缺失或冲突 / ❓ 代码层无法核实（需人工确认）

---

## 一、网站 / 落地页（决定过审速度）

- [ ] 有独立官网/落地页，而非只拿商店列表页当门面
- [ ] 公开可访问，首页前无登录墙
- [ ] 30 秒看懂：这是什么、解决什么、卖什么
- [ ] 产品已 live（已上架或有可下载/可试用入口），非"即将上线"
- [ ] 底部或显著处能找到：客服邮箱、隐私政策、服务条款入口

## 二、法律页面（强制，缺一不可）

- [ ] 隐私政策 Privacy Policy 已上线、可直接访问
- [ ] 服务条款 Terms of Service 已上线、可直接访问
- [ ] 隐私政策与 Chrome Web Store 后台"数据使用声明"内容一致（高频打回点）
- [ ] 退款政策清晰可见（Creem 专门审 refund policy）
- [ ] 法律页主体信息与 KYC 提交主体一致

## 三、客服可达性（强制）

- [ ] 使用品牌化客服邮箱（support@产品域名，非 gmail/qq 等通用邮箱）
- [ ] 该邮箱在公开网站上可见
- [ ] 该邮箱在用户后台 / 扩展设置页里也能找到（**两处都要有**）
- [ ] 邮箱真实可收件，承诺 3 个工作日内回复

## 四、定价与订阅（SaaS 必查）

- [ ] 定价清晰展示：价格、计费周期、币种明确
- [ ] 用户可自助取消订阅（Cancel Subscription API 或跳转 Creem Customer Portal）
- [ ] 取消入口在扩展内或后台真实可达，不是只留邮件申请
- [ ] 免费试用 / 退款条件写清楚，避免"诱导订阅"嫌疑

## 五、Manifest V3 / 扩展特有风险项

- [ ] permissions / host_permissions 最小化，无用不到的宽权限
- [ ] 隐私政策明确写出：收集哪些数据、存哪、是否上传、是否给第三方
- [ ] 仅本地存储（chrome.storage、不外传）→ 在隐私政策写明（加分项）
- [ ] 有上传到自己服务器/第三方 API → 写清用途、范围、保留期限
- [ ] content scripts 注入的页面范围与隐私政策描述一致（**注意 matches 是否随生产构建收敛**）
- [ ] 扩展图标/名称/描述与官网品牌一致（防"套壳/钓鱼"误判）
- [ ] 涉及 AI 能力（调大模型 API）→ 按 Creem AI Wrapper Compliance 自查

## 六、内容合规

- [ ] 不属 Creem 禁止类目（不确定先带描述/截图发客服确认）
- [ ] 不是 API 转售（reseller）；若是，需另提供原支付商、拒付率截图、迁移原因
- [ ] 无虚假宣传、无诱导话术、无侵权内容

## 七、KYC / 身份验证准备

- [ ] 备好身份证或护照
- [ ] 法律实体 / 个人主体、所有权信息真实准确
- [ ] 用 Alipay 收款：姓名用拼音，名姓之间留空格
- [ ] 提交信息与网站、法律页主体一致

## 八、提交前最后一遍

- [ ] 客服邮箱（网站 + 后台）与提交信息一致
- [ ] 隐私政策 / 服务条款 可访问且与 CWS 声明一致
- [ ] 产品 live，有可访问的真实入口
- [ ] 取消订阅路径真实可用
- [ ] 权限与数据流在隐私政策里交代清楚

---

## 被打回怎么办

1. 看邮件里 flagged issues
2. 逐项修复
3. Balance → Payout Account → Request re-review 重新提交
4. 优先动：客服邮箱可达性、法律页面、定价/取消订阅入口（高频打回点）
